# Client scripts for test management

This directory contains scripts that use the REST API to manage test terms and test templates. These scripts are intended for teachers, since it is more convenient to prepare tests locally and then sync them with Quixam rather than invent elaborate UI.

This folder is self-contained, so you may perform sparse checkout of `./tests-client` if you want to use only these scripts:

```bash
git clone --no-checkout https://github.com/krulis-martin/quixam.git
cd quixam
git sparse-checkout init --cone
git sparse-checkout set tests-client
git checkout main
```

This text explains how to use the scripts for test management:

- `login.php` -- performs authentication and saves the obtained token to a file
- `upload.php` -- uploads a test template (groups and questions) from a local directory
- `test-terms.php` -- uploads test terms from local CSV file

## Initialization and authentication

All main scripts are in the root of this folder and they all require `config.yaml` file present in the same directory. Use `config.yaml.example` as a template and fill in the required values, most importantly the URL of your Quixam instance.

All upload scripts need security token to operate, which is stored in a separate file (`.auth-token` by default, configurable in `config.yaml`). This token is generated by `login.php` script and refreshed every time one of the scripts is used. The login script uses parameters from `config.yaml`, prompts for the password and saves the obtained token to the file. Note, that the user must use **local** password, external authenticators (like CAS) do not work here.

The logout is not implemented, but you can simply delete the token file.

## Test templates

A test template instance must be created by the administrator beforehand in the database. The administrator will assign the **external ID** of the template and associate it with teachers who are the **owners**. Only owners can modify the template and add terms to it.

The contents of the test template (question groups and questions) are synced using the `upload.php` script. The script takes one argument -- the path to the test template `.yaml` file. The yaml file holds the main configuration (groups, points), while the individual questions are stored in individual `.md` files ([with additional custom tags](samples/README.md)). The additional files must be placed in the same directory as the `.yaml` file, each group has a folder (with the same name as the group) and all its questions are stored there. An example of the complete test can be found in [samples/startrek](samples/startrek) directory, which can be uploaded as:

```bash
php ./upload.php ./samples/startrek/config.yaml
```

(assuming `startrek` template is entity is already created and you are one of its owners).

The structure of the test template `.yaml` file is as follows:

```yaml
id: <template's-external-id>
groups:
  <group-name-1>: <count>
  <group-name-2>: <count>
  ...
```

The `<group-name-X>` are the identifiers of the groups and they must match precisely the directory names which must be located in the same directory as the `.yaml` file. The `<count>` values, specify how many questions from the group will be used in the test (when a test instance is generated from the template). The number should be a positive integer that do not exceed the number of questions (i.e., files) in the group.

## Test terms

Once a test template is properly filled with questions, exam terms can be created from it. The terms are managed by `test-terms.php` script which takes list of terms in a CSV file and synchronizes them with the Quixam instance. The CSV file must use commas (`,`) as separators and the first line must be a header with the following columns (the names must match exactly, the order is not important):

- `externalId` (string, required) -- the external ID of the term, must be unique across all terms in the system
- `location` (string, recommended) -- the room/building/address where the term will take place
- `scheduledAt` (datetime, recommended) -- the date and time when the term is scheduled (format is described below)
- `supervisors` (string, recommended) -- space-separated list of supervisors' emails or IDs
- `note_en` (string, optional) -- note in English
- `note_cs` (string, optional) -- note in Czech

The `scheduledAt` column can use anything that can be passed to the PHP `DateTime` constructor. The recommended formats are:

- `2026-03-25 13:42` (almost ISO format, but with space instead of `T` and without seconds)
- `14.3.2026 15:00` (European format with 24h time notation)
- `@1774394280` (Unix timestamp, note it must be prefixed with `@` to be recognized by the parser)

The supervisors can be identified by their email or by their external ID. If the string is successfully parsed as an email, it will be treated as such, otherwise it will be treated as an external ID.

Only the `externalId` column is required for the system to work properly, but the providing the recommended columns will make the data more readable by the users.

An example of the CSV file is provided in [samples/terms.csv](samples/terms.csv). You can use it as a template and upload it as:

```bash
php ./test-terms.php startrek ./samples/terms.csv
```

Please note, that the terms can be updated (synced) only until they are started or archived.
